# DECISIONS.md

## 为什么这样做（关键决策记录）

## 1. 先做「本地可跑」而不是先强依赖官方 CLI

**决策**：优先用 Bun + 本地脚本跑通核心链路。

**原因**：
- 当前仓库是还原代码，不是完整官方开发仓
- 先保证业务能力可验证，比先恢复完整 CLI 价值更高

---

## 2. 文件即数据层（.story-project）

**决策**：用 `.story-project` 作为统一资产目录。

**原因**：
- 透明、易审计、便于 Git 管理
- 方便 UI/脚本/工具共享同一数据源
- 后续迁移数据库成本可控

---

## 3. 增量演进：Commit 1~4 + UI v1~v5

**决策**：分阶段迭代，不一次性大改。

**原因**：
- 每阶段可验证，失败可回滚
- 降低复杂度，适合当前源码恢复环境

---

## 4. 导出格式必须覆盖生产交付

**决策**：最终支持 `json/md/csv/zip`。

**原因**：
- 不同角色需要不同格式
- zip 作为完整交付包是必须项

---

## 5. 先做 UI 运营面板，再做高级算法

**决策**：先做可视化流程与运维能力（状态、重试、下载、模板）。

**原因**：
- 团队协作效率提升最快
- 先把“可用性”做起来，再继续优化“智能性”
