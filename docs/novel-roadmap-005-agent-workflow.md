# 小说引擎多 Agent 工作流设计 v0.1

## 目标

将小说生成拆分成多个可协作、可审校、可扩展的 Agent 角色，让系统能够稳定支持：

- 短篇到超长篇
- 多题材写作
- 大纲 / 细纲 / 章节生成
- 连续性维护
- 小说到分镜的结构化输出

---

## 设计原则

1. **职责分离**
   - 每个 Agent 只负责自己最擅长的环节
   - 不让单个模型承担全部工作

2. **总控统一**
   - 需要一个主 coordinator 统一调度
   - 保证创作方向、节奏和商业目标一致

3. **结构优先**
   - 先生成结构，再填充正文
   - 先生成可审校的中间产物，再产出最终章节

4. **可回溯**
   - 每一步都要落运行记录
   - 每一步都要能回查输入和输出

5. **可持续长篇**
   - 需要内建摘要、记忆、状态维护机制
   - 不依赖单次上下文窗口

---

## 推荐的 Agent 角色

### 1. Market Analysis Agent
职责：
- 分析题材趋势
- 总结热门结构
- 提供读者画像建议
- 输出商业方向建议

输入：
- 题材偏好
- 热门样本摘要
- 用户目标

输出：
- 题材建议
- 节奏建议
- 结构建议
- 风格建议

---

### 2. Chief Editor Agent
职责：
- 确定作品定位
- 审查整体方向
- 控制内容边界
- 保证商业性与可持续性

输入：
- 市场分析结果
- 用户设定
- 作品目标

输出：
- 作品定位
- 创作策略
- 内容约束
- 风格基调

---

### 3. Worldbuilding Agent
职责：
- 构建世界观
- 维护规则体系
- 设计势力、地点、科技/能力系统

输入：
- 作品定位
- 题材方向

输出：
- 世界观摘要
- 规则集
- 势力图
- 地点图
- 时间线锚点

---

### 4. Character Agent
职责：
- 创建角色卡
- 维护人物关系
- 设计成长线
- 追踪角色状态变化

输入：
- 世界观
- 作品定位

输出：
- 主角 / 配角角色卡
- 关系图
- 动机与冲突
- 成长弧线

---

### 5. Outline Agent
职责：
- 生成总纲
- 生成卷纲
- 生成章纲
- 规划剧情阶段与爆点

输入：
- 世界观
- 角色
- 作品定位

输出：
- 总纲
- 卷纲
- 幕纲
- 章纲
- 章节目标

---

### 6. Chapter Agent
职责：
- 按章纲生成细纲
- 生成章节正文草稿
- 维护本章事件流

输入：
- 章纲
- 角色状态
- 历史摘要

输出：
- 章节正文
- 场景列表
- 事件列表
- 本章钩子

---

### 7. Continuity Reviewer Agent
职责：
- 检查逻辑矛盾
- 检查时间线
- 检查伏笔回收
- 检查角色行为一致性

输入：
- 章节结果
- 事件流
- 时间线
- 角色状态

输出：
- 修改建议
- 矛盾列表
- 修订要求

---

### 8. Style Unifier Agent
职责：
- 统一文风
- 统一节奏
- 统一叙述视角
- 统一术语使用

输入：
- 章节正文
- 风格基准

输出：
- 风格统一后的章节文本
- 风格偏差反馈

---

### 9. Storyboard Agent
职责：
- 把章节转换成分镜输入
- 提取视觉场景
- 给出镜头提示

输入：
- 章节结构
- 场景列表
- 角色状态

输出：
- 分镜脚本
- 画布节点输入
- 视觉关键词
- 镜头建议

---

## Coordinator 设计

主 coordinator 的职责是：

- 接收用户目标
- 拆解任务
- 调度各个 Agent
- 收集结果
- 合成最终输出
- 决定下一步写作方向

它不负责写全部正文，而是负责控制全局。

---

## 推荐工作流

### 流程 1：新建作品
1. Market Analysis Agent
2. Chief Editor Agent
3. Worldbuilding Agent
4. Character Agent
5. Outline Agent
6. 保存作品状态

### 流程 2：生成章节
1. Chapter Agent
2. Continuity Reviewer Agent
3. Style Unifier Agent
4. 保存章节版本
5. 更新角色 / 事件 / 时间线 / 伏笔

### 流程 3：章节转分镜
1. Storyboard Agent
2. 输出分镜脚本
3. 写入画布输入

---

## 与 restored-src 的借鉴关系

可以参考 `restored-src` 中的这些理念：

- coordinator / worker 模式
- 任务结果可追溯
- 会话恢复与状态管理
- 工具调用可组合
- 复杂任务分步执行

在小说引擎里，这些思想可以转化为：

- 主 coordinator + 多 worker 小说 Agent
- 作品状态持续保存
- 章节级任务链
- 审校与回滚机制

---

## Agent 间的数据交换

建议所有 Agent 的输入输出都采用结构化对象，而不是纯文本。

例如：
- `NovelProjectContext`
- `WorldbuildingState`
- `CharacterState`
- `OutlineState`
- `ChapterDraft`
- `ReviewReport`
- `StoryboardInput`

这样可以：
- 降低信息丢失
- 提高可审校性
- 方便前后端统一
- 方便后续存储与导出

---

## 长篇写作建议

百万字级别长篇不能靠单轮生成。

应采用：
- 总纲分层
- 卷级摘要
- 角色状态快照
- 时间线快照
- 伏笔栈
- 章节版本管理

并在每个阶段都让 review agent 做一致性检查。

---

## 任务调度策略

### 线性任务
适合：
- 开篇规划
- 章节生成
- 审校

### 并行任务
适合：
- 世界观与角色设定
- 不同卷的结构探索
- 多候选标题 / 开篇策略

### 迭代任务
适合：
- 章节修订
- 风格统一
- 连续性修补

---

## 失败处理

如果某一步 Agent 失败，应该：

1. 保留失败输入
2. 保留失败输出
3. 记录错误原因
4. 允许继续修订
5. 必要时回滚到上一版本

---

## 输出链路

最终小说工作流应当能够输出：

- 小说正文
- 章节摘要
- 场景切分
- 分镜提示词
- 角色状态变化
- 时间线变更
- 画布输入节点
