# 小说引擎实现顺序 v0.2

## 目标

把小说引擎从设计落到实现时，按最小风险、最高复用、最长可扩展性的顺序推进，避免一开始就陷入“生成全文”这种高风险目标。

> 说明：当前实现已完成第一批基础闭环（项目、世界观、角色、大纲、章节、运行历史、工作台接入）。本文件从“计划”升级为“按阶段迭代的执行顺序”，后续以新增进度日志跟踪每次落地。

---

## 当前状态面板

- **已完成**：小说规划总文档链、后端小说路由、数据模型、基础 CRUD、coordinator/worker 骨架、运行记录、工作台多面板、`plan` 基础闭环
- **进行中**：从“种子生成”升级为“多阶段结构扩展管线”
- **下一步**：补可编辑表单、引入多层级大纲、升级连续性/审校、开始市场偏好信号接入
- **记录方式**：每次实际落地同步更新 `novel-roadmap-009-progress-log.md`

---

## 总体原则

1. **先结构，后能力**
   - 先把数据模型、存储、API 约定定下来
   - 再接多 agent 编排
   - 最后再做正文生成和长篇优化

2. **先最小闭环，后完整链路**
   - 先做一个能创建作品、生成大纲、生成章节的闭环
   - 再扩展到审校、分镜、画布、漫剧

3. **先稳定可追溯，后智能增强**
   - 所有动作都要留下运行记录和版本记录
   - 再逐步增强市场分析、热门吸收、风格迁移能力

4. **先复用现有架构，再补新能力**
   - 参考 `restored-src` 的 coordinator / worker / task 模式
   - 不把小说引擎做成另一个孤立系统

## 风险提示

1. **不要先做全文生成**
   - 直接生成长篇正文会让结构失控

2. **不要把所有逻辑塞进一个 Agent**
   - 必须拆分角色与职责

3. **不要忽视摘要层**
   - 长篇必须依赖摘要和状态快照

4. **不要忽视版本化**
   - 作品、角色、章节、大纲都要可回滚

5. **不要脱离下游链路**
   - 小说引擎最终是为分镜和漫剧服务的
